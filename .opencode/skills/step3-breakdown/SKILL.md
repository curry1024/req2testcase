# Step 3: 功能点分解

将 Step2 的结构化需求拆解为原子级可测功能点，生成 XMind 脑图供用户审核确认。

## 为什么需要确认

- LLM 对需求的理解可能存在偏差
- 用户可能有额外的业务知识需要补充
- 确认后的功能点才进入用例生成，避免批量返工

## 输入

从 `.opencode/work/<工作目录>/step2_requirements.json` 读取结构化需求。

## 输出

产出两个文件，写入同一工作目录：

| 文件 | 用途 |
|------|------|
| `step3_features.json` | 机器可读的功能点数据，供 Step4 使用 |
| `step3_features.xmind` | XMind 脑图，供用户在 XMind 中打开审阅 |

## 工作流程

1. 读取 `step2_requirements.json`，获取所有 REQ
2. 对每条 REQ，拆解为逻辑测试场景/分支：
   - **状态分支**：根据 if/else 条件拆分不同状态路径（如 显示/隐藏、上架/下架）
   - **数值区间**：识别数值型条件拆分区间（如 <10、10~15、≥16）
   - **异常分支**：识别错误处理、失败路径
3. **XMind 只记录分支逻辑，不写入测试数据。** 节点上只写场景描述，前置条件/测试值/预期结果留给 Step4
4. 将功能点写入 `step3_features.json`
5. 调用 `scripts/generate_xmind.py` 根据 JSON 生成 XMind 文件
6. **暂停，请用户打开 XMind 审核确认后，再继续 Step4**

## 分类说明

| category | 说明 | 示例 |
|----------|------|------|
| `state` | 状态/条件分支 | 不显示/显示未解锁/解锁 |
| `range` | 数值区间分支 | 进度<10、10~15、≥16 |
| `exception` | 异常/错误路径 | 密码错误、断网超时 |
| `flow` | 正常流程 | 第1场成功进第2场 |

## 输出 Schema (JSON)

```json
{
  "$schema": "feature-points-v1",
  "meta": {
    "source_file": "原始文件名",
    "total_requirements": 3,
    "total_features": 9
  },
  "features": [
    {
      "id": "FP-001",
      "req_id": "REQ-001",
      "module": "活动入口",
      "name": "入口显示与解锁条件-不显示",
      "category": "state",
      "description": "进度<10关时入口不显示",
      "priority": "P1"
    }
  ]
}
```

## 字段说明

| 字段 | 说明 |
|------|------|
| `id` | FP-001 自增编号 |
| `req_id` | 关联回 Step2 的 REQ，保持追溯链 |
| `module` | 从父 REQ 继承 |
| `name` | 功能点名称，格式：`需求名-场景/分支` |
| `category` | 分支类型：`state`(状态分支)、`range`(数值区间)、`exception`(异常)、`flow`(流程) |
| `description` | 一句话描述这个测试场景，不写用例细节 |
| `priority` | 按漏测风险判定，见下方规则 |

## 优先级判定规则（按漏测风险，不按分类类型）

| 优先级 | 判定标准 | 示例 |
|--------|----------|------|
| P0 | 核心主流程的边界临界值，漏测等于功能缺失 | 每个状态至少 1 条 P0 |
| P1 | 重要场景的常规值、非边界状态 | 状态的中间值、次边界 |
| P2 | 低频场景、极端值、次要变化 | 极限高值、罕见异常 |
| P3 | 几乎不会出现的边缘场景 | 低概率并发、极端网络 |

> **P0 分配原则**：一个功能有 N 个状态分支，每个分支的边界临界值至少分配 1 条 P0。例如 3 个状态（不显示/显示未解锁/解锁），至少 3 条 P0（9/10/16），其余测试值为 P1，P2 留给极小概率场景。

> 注意：Step3 只拆分逻辑分支，不填测试数据、不写用例步骤。那些是 Step4 的工作。

## XMind 脑图结构

XMind 只展示需求 → 测试场景的**逻辑分支树**，不写入用例细节：

```
需求文档名 (根节点)
├── REQ-001: 入口显示与解锁条件
│   ├── 不显示 (进度<10关)
│   ├── 显示未解锁 (进度10~15关)
│   └── 解锁 (进度≥16关)
├── REQ-002: 入口状态切换
│   ├── 未开始 (显示"开始"+红点)
│   ├── 进行中 (显示倒计时)
│   ├── 活动结束
│   └── 已完成
└── ...
```

节点上只写场景描述，不写前置条件/测试值/预期结果。让用户快速扫一眼，确认逻辑覆盖是否完整。

## 注意事项

- 结合需求原文的 if/else 条件和数值区间做拆解
- **数值型条件必须拆出区间**：如"<10 → 不显示, 10~15 → 显示未解锁, ≥16 → 解锁"
- **枚举型需求必须每种都拆**：如"5种任务类型"→至少要拆 5 个 FP，每种类型一个；"7天"→ 至少要拆 3 个 FP（Day1首天/Day7末天/Day4中间天）
- **操作类需求不重复拆状态**：如果操作依附于其他 REQ 的状态（如"点击跳转"依附于"状态切换"），操作不另拆测试点，并入对应状态的用例
- **禁止拆组合条件 FP**：如果需求写"A AND B → C"，不要单独拆一个"满足 A 且 B"的 FP。A 和 B 各拆一个 FP 分别测正反，C 的验证在 Step4 中跨 FP 组合即可
- **禁止为同一对象的属性各拆 FP**：如果表 A 定义类型、表 B 定义同类型的计数规则，不要拆成两组 FP。计数规则归入类型 FP。标准：两张表描述同一个对象的行为 → FP 合并
- **枚举变体若操作路径不同，必须原子化拆 FP**（与上一条不冲突）：当一个枚举的每个取值对应**不同的执行操作**（如通关的 普通/故事/突发事件 模式、金币的 关卡/广告/转盘/充值/活动 渠道、道具的 灯泡/橡皮/座位 + 购买不计入、广告的 激励/插屏 + 免券不计入），**每个取值拆成一个独立 FP**，不要合并成一个"各模式/各渠道"的笼统 FP。
  - 判据：取值差异来自"做的操作不同" → 各自原子 FP；来自"同一操作下的不同属性" → 合并到一个 FP。
  - 反向规则（不计入/重置类）也各自成 FP：如"购买道具不计入""免广告券不计入""连胜中断重置"。
- **同一 REQ 下 FP 组命名必须区分测试目的**：如果合并后 FP 数量 >8 且内容明显分属两类场景（如"类型定义"vs"计数规则"），拆出两个子模块名（如"任务计数规则"和"任务类型定义"），不要让所有 FP 挂同一个 feature_point 名
- 一个 REQ 通常拆出 2~8 个测试场景
- 不要在图里写用例内容（测试步骤、测试数据等），那是 Step4 的工作
- **优先级按漏测风险判定，不要用分类类型自动推导**。同一功能的所有状态分支至少 P1
- **需求模糊时**：在 XMind 节点末尾加 `?` 标记（如"已完成（跨天处理？）"），并在 JSON 的 description 中写 `[待确认]`
- 用户审核确认前，不要启动 Step4

## 分解后自检

生成 FP 清单后，逐条 REQ 检查以下覆盖维度：

| 检查项 | 通过标准 |
|--------|---------|
| 状态分支 | 每种状态至少 1 个 FP |
| 数值区间 | 每个区间的上下边界各 1 个 FP |
| 枚举类型 | 每种枚举值（任务类型/天数/语言等）至少 1 个 FP |
| 异常路径 | 每种错误/异常至少 1 个 FP（不满足条件、失败、超时等） |
| UI 组件 | 需求中提到的每个 UI 组件（图标/按钮/进度条/Tab）都有对应的状态覆盖 FP |
