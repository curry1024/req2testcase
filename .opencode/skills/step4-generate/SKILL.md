# Step 4: 用例生成

根据确认后的功能点，逐条生成详细测试用例，产出 JSON 和 Excel 预览供用户审核。

## 输入

从 `.opencode/work/<工作目录>/step3_features.json` 读取已确认的功能点。

## 输出

| 文件 | 用途 |
|------|------|
| `step4_testcases.json` | 机器可读的用例数据，供 Step5 使用 |
| `step4_testcases.xlsx` | Excel 预览，供用户打开审核 |

## 工作流程

1. 读取 `step3_features.json`，获取所有已确认的测试场景
2. **去重检查**：扫描所有测试场景，识别跨 REQ 的重复覆盖
3. 对每个测试场景生成用例：
   - **数值区间场景**：用边界值法拆解，每条 FP 至少 3~4 条用例
   - **状态分支场景**：每种状态至少 1 条用例，包含状态内关键操作
   - **操作类场景**（点击跳转、按钮行为）：不重复验证已有状态，只测操作本身的跳转行为
4. 将用例写入 `step4_testcases.json`
5. 调用 `scripts/generate_excel.py` 生成 `step4_testcases.xlsx`
6. **暂停，请用户打开 Excel 审核确认后，再继续 Step5**

## 跨测试点去重规则（必须遵守）

当一个操作依赖于其他测试点已覆盖的状态时，**直接复用该状态的用例，不要另起一套**：

```
示例：
  REQ-002 覆盖了入口的 4 个状态（未开始/进行中/结束/完成）
  REQ-003 是"点击入口跳转到不同界面"
  
  错误做法：REQ-003 重新生成 4 条用例，重新描述 4 个状态
  正确做法：在 REQ-002 的每条状态用例里加一个步骤——"点击入口，验证跳转正确"
```

### 判断标准

- 如果操作对象的状态已经被另一个测试点覆盖 → **合并操作到已有用例**
- 如果操作本身有独立的状态机/分支 → 独立拆用例
- 一个测试步骤不要在两套用例里验两遍

## 边界值分析规则（必须遵守）

当测试场景涉及**数值区间**时，必须用边界值法拆出具体的测试值：

### 区间型边界

```
需求: 关卡<10 不显示, 10~15 显示未解锁, ≥16 解锁
  
不显示 →  测试值: 5(常规), 9(上边界), 10(超边界)
显示未解锁 →  测试值: 10(下边界), 12(常规), 15(上边界)
解锁 →  测试值: 15(未解锁上界), 16(下边界), 17(常规), 40(高值)
```

### 三区间模板

| 区间 | 测试值选取 |
|------|-----------|
| <下限 | 常规、上边界、上边界+1 |
| [下限, 上限] | 下边界、常规、上边界 |
| >上限 | 上边界、下边界常规、高值 |

### 状态切换边界

``` 
- 从X状态切到Y状态时的临界值
- 刚满足条件时的表现
- 刚好不满足条件时的表现
```

## Excel 表头

| 列 | 字段 | 说明 |
|----|------|------|
| A | 用例编号 | TC-001, TC-002... 自增 |
| B | 模块 | 所属模块名 |
| C | 测试点 | 对应 REQ 的功能名（如"入口显示与解锁"），相同测试点合并居中 |
| D | 用例标题 | 简洁概括场景（如"入口不显示-关卡小于10"），不写"验证"前缀 |
| E | 前置条件 | 执行用例前需满足的条件 |
| F | 测试步骤 | 编号步骤，如 "1.打开登录页\n2.输入账号密码\n3.点击登录" |
| G | 预期结果 | 期望看到的结果 |
| H | 优先级 | P0 / P1 / P2 / P3 |
| I | 测试结果 | 预留填写：P(Pass) / F(Fail) / N/A |
| J | 备注 | 其他补充说明 |

## 输出 Schema (JSON)

```json
{
  "$schema": "testcases-v1",
  "meta": {
    "source_file": "原始文件名",
    "total_features": 15,
    "total_testcases": 45
  },
  "testcases": [
    {
      "id": "TC-001",
      "feature_point": "入口显示与解锁",
      "feature_id": "FP-001",
      "module": "活动入口",
      "title": "入口不显示-关卡小于10",
      "preconditions": ["用户已注册", "账号状态正常"],
      "steps": ["打开登录页", "输入账号 test@test.com", "输入密码 Abc12345", "点击登录按钮"],
      "expected_results": ["页面跳转至首页", "标题栏显示用户昵称"],
      "priority": "P0",
      "test_result": "",
      "notes": ""
    }
  ]
}
```

### ⚠️ 字段类型契约（必须遵守，违反会导致渲染崩坏）

| 字段 | 类型 | 反例后果 |
|------|------|---------|
| `preconditions` | **JSON 数组**（字符串列表） | 写成字符串会被逐字符遍历，每个字带一个项目符号 |
| `steps` | **JSON 数组**（每步一个元素） | 写成字符串会被逐字符编号 |
| `expected_results` | **JSON 数组**（每个检查项一个元素） | 写成字符串会被逐字符编号；单数 `expected_result` 已弃用 |

- 即使只有一条，也要写成单元素数组：`"expected_results": ["xxx"]`。
- 统一用复数 `expected_results`，不要用单数 `expected_result`。

### 边界值扩展（适用于所有含数值的 FP）

**不仅 range 类型需要边界分析。** 只要 FP 的 description 中提到具体数值（阈值、档位、目标值），即使 category 是 state/flow，也必须拆出边界用例：

| 数值类型 | 扩展规则 | 示例 |
|---------|---------|------|
| 档位阈值 | N-n, N, N+n | 积分300→ 测 299(未达标), 300(刚好), 301(达标后) |
| 目标值 | 0, 目标-1, 目标, 目标+1 | 通关3关→ 测 0关, 2关, 3关, 4关 |
| 天数范围 | Day边界 | Day1(首天), Day7(末天), 跨天时刻(23:59→0:00) |
| 百分比/区间 | 下界-1, 下界, 上界, 上界+1 | "50%-70%已解锁建筑" → 测 49%, 50%, 70%, 71% |

### 枚举型 FP 的遍历规则

当 FP 涉及"每种类型/每一天/每个状态"的枚举时，**不能只测1-2个采样**：

- **任务类型枚举**（如 5 种任务类型）：至少每种类型 1 条正例 + 1 条反例
- **天数枚举**（Day1~Day7）：至少测 Day1(首)、Day7(末)、Day4(中) + 跨天边界
- **状态枚举**（如 3 种礼盒状态）：每种状态至少 1 条
- **多语言枚举**（3 种语言）：至少每种语言 1 条正例 + 1 条切换用例

**遍历原则**：同一个测试点下，所有枚举类型必须采用**统一结构**遍历。
反例：5种任务类型，只给其中2种写了详细子用例，另外3种一笔带过。
正例：计数值中每种类型(通关/连胜/金币/道具/广告)各1条；类型定义中每种类型×每种完成条件各1条。

### 预期结果拆分规则

**一个检查项 = 一行**，不要把多个检查项塞进同一个单元格用编号分隔。

反例（单单元格塞多条）：
```
预期结果: "1. Day1进度3/3 2. Day2进度3/5 3. 锁定天偏灰"
```

正例（一个检查项一行，复用同一条用例的其他字段）：
```
行1: 预期结果: "Day1进度3/3满格"
行2: 预期结果: "Day2~7各天进度同步显示3/各自目标值"
行3: 预期结果: "锁定天进度条偏灰"
```

拆分标准：
- 涉及**不同天/不同状态**的检查 → 各成一行
- 涉及**不同对象**的检查（如A进度+B进度） → 各成一行
- **视觉行为分离**（如数值 + 颜色） → 各成一行
- 一个单元格内需要"并且""同时"连接多件事的 → 拆成多行

> **为什么拆行**：导出时一条用例的步骤/前置合并、但**每行预期独立判 P/F 并可单独提 BUG**。例如检查一幅画的"大小/图形/颜色"，颜色错了只判颜色 Fail+提单，大小图形照样 Pass。

### ⚠️ 拆"用例" vs 拆"预期行"——别搞混

| 情况 | 处理 | 例 |
|------|------|----|
| **同一组步骤**，要验证多个维度 | **同一条用例**，预期拆成多行 | 看一幅画 → 大小/图形/颜色 三行预期 |
| **不同的操作步骤**（不同模式/渠道/类型） | **拆成多条独立用例**，各自有步骤 | 通关-普通模式 / 通关-故事模式 / 通关-突发事件模式 各一条 |

判据：**预期行的差异来自"看的角度不同"→拆行；来自"做的操作不同"→拆用例。**
反例：把"普通模式通关计入 / 故事模式通关计入 / 突发事件模式通关计入"塞进一条用例的三行预期——但步骤只写了"普通模式通关"，另外两种根本没执行。必须拆成 3 条用例。

### 组件生命周期覆盖

当一个 UI 组件存在多状态切换时，必须覆盖**端到端生命周期**：

需求示例：任务行右侧图标有"灰色积分 → 🔒 → 领取按钮 → ✅勾"共 4 种状态。

| 覆盖维度 | 说明 |
|---------|------|
| 状态切换触发 | 从每个状态到下一个状态的临界条件 |
| 跨天状态保持 | 退出后重登，状态是否保持 |
| 同一天多任务交叉 | A 任务已领取 + B 任务待领取的共存表现 |
| 不可逆状态 | 已领取→不可退回（点击无反应） |

### 跨表格数据交叉验证

当 step1 中有多张关联的表格数据时，必须在一条用例中联合验证：

需求示例：积分值表 + 三档奖励表 + 目标值表

```
TC-XXX: Day7 5任务全完成 → 日满分800 → 总积分达3000 → 三档礼盒全部可领 + 礼盒内容逐项验 + 积分不扣减
```

而不是把"积分发放""礼盒领取""积分扣减"拆成三条互不关联的用例。

## 生成前自检清单

生成用例后，**必须逐项确认**：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | step1 数据完整性 | 所有表格/段落内容已提取，无 `[MISSING_DATA]` 残留 |
| 2 | FP 覆盖率 | 每条 FP 至少 1 条用例，range 型 ≥3 条 |
| 3 | 枚举遍历 | 任务类型/天数/状态/语言等枚举至少各 1 条 |
| 4 | 边界值 | 所有数值阈值（档位/目标值/天数）均已测边界 ±1 |
| 5 | 组件生命周期 | UI 组件完整状态切换路径已覆盖 |
| 6 | 跨表交叉 | 关联表格数据的联合验证已生成 |
| 7 | 红点/提醒 | 触发条件 × 红点位置的交叉组合已覆盖 |
| 8 | 数据持久化 | 关键状态（已领取/已解锁/积分/礼盒状态）在重登后保持 |

发现未通过时，补充用例直到清单全部通过，再生成 Excel。

## 生成规则

- 用例编号 TC-001 起自增，即使跨模块也不重置
- `feature_point` 填对应 REQ 的功能名（如"入口显示与解锁"），多个场景共用
- `title` 简洁概括本条用例的场景（如"入口不显示-关卡小于10"），不要写"验证""测试"等前缀
- 测试步骤必须编号，每条步骤一行，用换行符分隔
- 预期结果需明确可验证，避免"正常""成功"等模糊表述
- 涉及数值区间的用例，必须给出具体测试值，覆盖边界点
- **需求不明确时**：不要编造答案，在 `notes` 中写入 `[待确认] 具体疑问`，让 QA 评审时确认

## 需求不明确的处理

当生成用例时发现需求文档有歧义或遗漏：

1. 先在 `notes` 写入问题（如 `[待确认] 活动持续时间是自然天还是滚动24小时？`）
2. 按最常见的理解继续生成用例（不中断）
3. 生成完汇总所有 `[待确认]` 项，提醒用户检查

常见场景：
- 时间计算不明确（时区、起止点、精度）
- 数值范围不明确（取整方式、边界包不包含）
- 交互跳转不明确（弹窗/页面/刷新）
- 状态切换触发条件不明确

## 注意事项

- 用户审核确认前，不要启动 Step5
- Excel 中测试结果列默认为空
- 用 `openpyxl` 库生成 `.xlsx`，脚本路径：`scripts/generate_excel.py`
- **Excel 格式要求**：所有单元格统一使用左对齐 + 垂直居中（horizontal=left, vertical=center），表头可保持居中但正文数据区必须左对齐垂直居中
