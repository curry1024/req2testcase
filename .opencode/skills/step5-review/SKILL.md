# Step 5: 用例审查

对 Step4 生成的测试用例进行自动化质检，发现明显问题，输出审查报告。

## 输入

| 文件 | 用途 |
|------|------|
| `step3_features.json` | 功能点清单，用于覆盖度对比 |
| `step4_testcases.json` | 生成的测试用例 |

## 输出

| 文件 | 用途 |
|------|------|
| `step5_review_report.md` | 审查报告，列出所有问题和建议 |

## 工作流程

1. 读取 step3 和 step4 的 JSON 文件
2. 运行 `scripts/review.py` 进行规则检查
3. LLM 读取规则检查结果 + 用例内容，进行语义检查
4. 汇总生成 `step5_review_report.md`
5. **暂停，请用户查看审查报告，决定是修改用例还是继续输出**

## 审查维度

### 规则检查（scripts/review.py）

| 检查项 | 规则 | 严重度 |
|--------|------|--------|
| 覆盖度 | Step3 的功能点是否全部有对应用例 | 严重 |
| 完整性 | 标题、步骤、预期结果是否为空（预期结果兼容 `expected_results`/`expected_result`，二者有其一即可） | 严重 |
| 编号连续性 | TC-001, TC-002... 是否连续无跳号 | 轻微 |
| 步骤过于简略 | 有效步骤数 < 2（单字符串按换行拆分计数，列表按元素累加） | 轻微 |
| 步骤格式 | 是否存在空白步骤项（**不再强制内联编号**，编号由导出脚本自动添加） | 轻微 |
| 预期结果重复 | 同一用例的 `expected_results` 是否存在完全重复项 | 轻微 |
| 优先级分布 | P0/P1/P2/P3 占比是否合理 | 提示 |
| 模糊表述 | 预期/测试数据是否含 "正常""成功""有效值""任意" 等模糊词 | 提示 |

> 数据 schema 统一用复数 `expected_results`（数组）。`expected_result`(单数) 已弃用，仅为兼容老用例保留。

### LLM 语义检查

| 检查项 | 说明 |
|--------|------|
| 逻辑一致性 | 前置条件与测试步骤是否矛盾 |
| 预期可验证性 | 预期结果是否具体、可验证（避免"正常""成功"等模糊词） |
| 冗余检测 | 是否有标题/内容高度相似的重复用例 |
| 边界遗漏 | 基于功能点描述，是否遗漏明显的边界/异常场景 |
| 模糊表述 | 测试数据中是否有 "有效值""任意值" 等模糊词 |

## 审查报告格式

```markdown
# 测试用例审查报告

**审查时间**: 2025-06-10 15:30
**用例总数**: 45
**功能点数**: 15

## 问题汇总

| 严重度 | 数量 |
|--------|------|
| 严重 | 2 |
| 轻微 | 5 |
| 提示 | 3 |

## 严重问题 (需修复)

### [覆盖缺失] FP-005 无对应用例
**关联功能点**: 订单创建-异常流程
**建议**: 至少补充 1 条异常用例

### [用例不完整] TC-012 缺少预期结果
**用例**: TC-012 验证支付超时处理
**字段**: expected_result 为空

## 轻微问题 (建议修复)

...

## 提示 (可选优化)

...
```

## 审查后的流程

- 如果存在严重问题：用户修改 Step4 用例后重新审查
- 如果只有轻微/提示问题：用户可选择忽略，继续最终输出
